# 我是 RepoPulse 🤖

我是一个住在飞书群里的 GitHub 热点发现与技术研究机器人。

简单说，我不是来群里刷存在感的，我是来帮技术团队更快看懂项目、少走弯路的。

你可以把我理解成一个“面向技术团队的项目雷达 + 研究助手”：

- 我会盯着 GitHub 上最近正在升温的项目 👀
- 我会告诉你哪些项目值得看、为什么值得看 📈
- 我也会在你点名某个仓库时，快速整理出“它是什么、适合谁、能不能试、怎么开始试”这类真正有用的信息 🧭

我的目标不是制造更多信息，而是帮技术团队更快做出第一轮判断：

- 这个项目值不值得继续关注？
- 它为什么突然火起来了？
- 我适不适合看它？
- 我今晚能不能把它跑起来试一试？

如果只用一句话介绍我：

> 我是一个在飞书里帮技术团队发现 GitHub 热点、理解项目价值、判断试玩门槛的机器人。🚀

---

## ✨ 我是来帮你省判断成本的

技术团队每天都会遇到这几种情况：

- GitHub 上项目很多，但很难快速判断哪些真的值得看。
- 看到某个项目突然很火，却说不清它为什么火、适不适合自己。
- 想试一个项目，但不知道上手门槛、最短路径和常见阻塞点。

我想解决的，其实就是这类“发现成本高、判断成本更高”的问题。

我希望把“偶然刷到一个项目”，变成一条更结构化的流程：

1. 先发现最近正在升温的 GitHub 项目。
2. 再判断这些热度到底是短期爆发，还是更值得持续关注的趋势。
3. 然后把结论整理成群里能快速读懂的摘要。
4. 如果你想继续看，我再补一份更完整的研究结果和试玩判断。

所以从角色上说，我更像一个长期在线、随叫随到的“技术情报入口”：

- 我面向的是技术团队，而不是泛内容推荐场景。
- 我关注的是“值得跟进的项目”，而不是“星最多的项目”。
- 我强调的是“帮助你判断”，而不是“替你做最终决策”。

## ✨ 我能帮你做什么

### 1. 我能给你日榜和周榜 📊

我会持续关注与 AI、开发工具、基础设施、前端 / 全栈等方向相关的项目，并输出两类榜单：

- **日榜**：更关注最近 24 小时里突然升温的项目
- **周榜**：更关注最近 7 天里持续增长、热度更稳的项目

它们回答的问题不太一样，你可以把它理解成：

- 日榜适合你快速看“今天有什么新东西冒出来了”
- 周榜适合你判断“这一周哪些项目是真的在持续变热”

一个看新鲜，一个看趋势。👀

### 2. 我能帮你查一个具体项目 🔍

如果你在群里点名某个仓库，我就可以直接开始整理它的研究结果。

你可以这样叫我：

- `/a openai/openai-python`
- `/a https://github.com/openai/openai-python`
- `/a openai python sdk`

你点名，我开查；不过在正式开始前，我会先确认你到底想查哪个仓库。

### 3. 我能告诉你“它是什么，为什么最近火” 💡

当我研究一个项目时，我不会只写一段空泛介绍。我会优先回答这些技术团队最常问、也最容易卡住的问题：

- 它到底是做什么的？
- 它为什么最近突然值得关注？
- 它更适合什么样的团队或开发者？
- 它不太适合什么场景？

这部分的目标，是帮你在很短时间里决定：这个项目值不值得继续往下看，还是可以先划走。✋

### 4. 我能帮你判断“能不能试，怎么开始试” 🛠️

这是我和很多“只会写项目简介”的机器人不太一样的地方，也是我最想做扎实的一层能力。

除了告诉你项目是什么，我还会尽量告诉你：

- 它能不能快速试玩
- 如果能试，最短的体验路径是什么
- 你需要先准备什么
- 跑到什么结果，才算真的成功
- 最常见的阻塞点是什么

简单说，我不只做“研究”，我也会提前帮你踩一脚刹车，或者踩一脚油门。

### 5. 我会把结果整理成适合团队传播的内容 🧾

我会把结果分成两层交付：

- **飞书群摘要**：适合快速读、快速讨论
- **飞书文档**：适合沉淀、转发、后续继续看

群里我会尽量只说重点，不把长文一股脑塞进去；如果你想看更完整的内容，我会把完整版放进文档里。

## 💬 你可以怎么叫我

我当前支持这些飞书命令：

| 用途 | 命令 |
| --- | --- |
| 分析项目 | `/a <repo|url|keyword>` |
| 分析项目 | `/analyze <repo|url|keyword>` |
| 触发日榜 | `/d [topN]` |
| 触发日榜 | `/daily [topN]` |
| 触发周榜 | `/w [topN]` |
| 触发周榜 | `/weekly [topN]` |
| 查看帮助 | `/h` |
| 查看帮助 | `/help` |

飞书群里默认仅在真实 `@机器人` 后处理命令；私聊可直接输入 slash 命令。

说明：

- 群聊支持 `@机器人 /help`、`@机器人 日榜 [topN]`、`@机器人 <repo|url|keyword>`
- 上面的 `@机器人` 只是占位符，请以群里的实际机器人显示名为准
- 默认使用飞书官方 SDK 长连接接收群消息；如需临时回退 HTTP 回调，可设置 `FEISHU_LONG_CONNECTION_ENABLED=false`
- 如需放开群聊里的裸 slash 命令，可设置 `FEISHU_GROUP_REQUIRE_BOT_MENTION=false`；该开关不会把群聊里的裸 `日榜` / 仓库关键词文本也一起放开

另外，为了让交互更自然，我在收到有效命令后会先加一个 `Typing` 表情，表示“收到，我已经开始干活了” ⌨️

## ⚙️ 我背后是怎么转的

### 1. 我先找候选，再决定谁值得上榜

我不会直接照搬某个现成榜单，也不会只看“谁的总 Star 最多”。

换句话说，我不是那种“看到数字大就先冲上去”的机器人。

我的做法更像是先建立一个候选池，再从里面筛出真正值得关注的项目。我会优先从几类信号里找候选：

- 最近仍在活跃更新的项目
- 最近 30 天里快速冒头的新项目
- 已经有一定体量，但最近 7 天还在继续升温的项目

这样做的好处是：

- 我不会只盯着头部大项目
- 我更容易发现那些刚刚开始被注意到的新东西
- 我能把“体量大”和“最近真的在升温”区分开来

### 2. 我会区分“今天很热”和“这一周都在升温”

我把日榜和周榜分开看，是因为它们要回答的问题根本不同：一个更像“今天谁冒头了”，一个更像“这周谁真的站稳了”。

#### 日榜里，我主要看这些 🔥

- 最近 24 小时的增长是不是明显
- 这种增长是不是相对体量来说也很突出
- 项目最近有没有继续更新代码
- 它是不是一个还很新的项目，正在快速试错
- 它更像是真实产品，还是模板 / 脚手架 / 清单仓库

所以，日榜不是“谁最大”，而是“今天谁最值得注意”。

#### 周榜里，我更在意这些 📈

- 最近 7 天是不是一直在往上走
- 热度是不是比较稳定，而不是只爆了一天
- 近 24 小时里是不是还在继续增长
- 项目热度和维护状态是不是一致

所以，周榜不是“谁今天最吵”，而是“谁这一周都值得持续关注”。

### 3. 我会尽量把“为什么上榜”讲清楚

我不希望榜单看起来像黑盒，更不希望你看完之后只剩一句“它怎么就排第一了？” 🤔

所以每个项目上榜时，我都会尽量给出相对具体的理由，比如：

- 最近增长主要来自哪里
- 它是短期爆发还是持续升温
- 它最近是否仍有代码更新
- 它是不是一个新项目

这样你看到榜单时，不只是知道“它排第几”，也知道“它为什么会排在这里”。

### 4. 当你点名一个项目时，我会先找一手证据

我在研究一个项目之前，不会直接把仓库名扔给模型，让模型自由发挥。

我会先从 GitHub 拿到一手信息，例如：

- 仓库简介
- README 摘要
- 最近的 release
- 最近的 commit
- 仓库根目录的一些关键路径

这些信息很重要，因为它们能帮我先建立“这个项目自己的事实基础”。在这个基础上，我再去补充公开网络资料，会比完全放开搜索更稳，也更不容易跑偏。

### 5. 我不只做研究，我还会做“试玩判断”

这是我现在最重要的一层能力升级。🚧

我不会满足于输出一篇“项目介绍”，我还会尽量把下面这些问题说清楚：

- 这个项目能不能很快试起来？
- 如果能试，最短路径是什么？
- 需要准备哪些前置条件？
- 什么结果才算真的跑起来了？
- 最容易卡住的地方是什么？

如果一个项目看起来很火，但实际上需要复杂环境、云资源、API Key、GPU 或很多隐含前置条件，我会尽量把这些门槛提前讲出来。

这样你就不会因为“看起来很酷”，结果花了半天装环境，最后发现当天根本没法试。😅

### 6. 我会把结果拆成“适合聊天”与“适合沉淀”两层

我知道群聊和文档不是同一种阅读场景：一个要快，一个要全。

所以我的交付方式是分层的：

#### 在群里，我会说重点 💬

我会优先给出这些内容：

- 它是什么
- 为什么最近值得看
- 能不能快速试玩
- 最短体验路径
- 主要风险或门槛
- 仓库链接和文档链接

#### 在文档里，我会说完整一点 📄

我会把更完整的内容沉淀进去，比如：

- 项目简介
- 为什么最近火
- 适合谁 / 不适合谁
- 是否能快速试玩
- 最短体验路径
- 前置条件与外部依赖
- 常见阻塞点
- 最佳实践
- 局限与风险
- 引用来源

这样既不让群消息过载，也不让研究结果只停留在一条很快就会被刷上去的聊天记录里。

### 7. 我会尽量保证结果靠谱，而不是只求“看起来像样”

为了让结果更可信，我会尽量把几件基础工作做扎实：

#### 榜单上，我尽量避免“瞎排”

- 我不是只看总 Star
- 我会区分短期爆发和持续趋势
- 我会考虑项目是否真的还在活跃推进
- 我会对模板、搬运、清单类项目做降权

#### 详情上，我尽量避免“硬编”

- 我优先用仓库自己的一手信息
- 我优先参考官方来源
- 有冲突时，我会尽量明确说冲突在哪里
- 信息不够时，我会明确告诉你“现在还不能可靠判断”

#### 运行上，我尽量避免重复折腾

- 榜单和详情都会做缓存
- 同一个项目不会被并发重复研究
- 已经有文档的项目会尽量复用已有文档
- 遇到可恢复的网络波动时，我会做有限重试

#### 工程上，我尽量不是“只会说”

我背后的主要链路有实际测试覆盖，这至少说明我不是一份“只写设计、不看实现”的说明文。✅

当然，测试覆盖并不等于没有边界；具体通过情况，还是应该以当前本地或 CI 的实际结果为准。

## 🚫 我不能做什么

为了不让你对我产生错误期待，我也需要先把边界说清楚。

### 1. 我不是万能的 GitHub 解释器

我擅长做“第一轮筛选”和“结构化理解”，但我不能替代深入源码阅读、真实接入验证或长期跟踪调研。

如果你已经进入“准备正式引入这个项目”阶段，那还需要工程团队继续做自己的验证。

### 2. 我不是实际执行仓库代码的评测平台

我会尽量告诉你怎么试、试到什么程度算成功、最可能卡在哪里，但这不等于我已经亲自把每个项目都真实跑了一遍。

所以我给你的更准确定位应该是：

- 高可信的试玩引导
- 而不是最终的运行担保

### 3. 我不是全网所有技术热点的总入口

我当前主要还是以 GitHub 为中心来发现和研究项目。

这意味着：

- 我更适合看开源项目趋势
- 不适合把我当成“所有技术新闻”的统一入口
- 我对非 GitHub 主导的热点覆盖会更弱

### 4. 我不是无限制的持续对话助手

我当前更擅长的是：

- 榜单推送
- 单次项目研究
- 试玩判断
- 文档沉淀

而不是长时间、多轮、上下文极深的持续问答式协作。

### 5. 我也不会替你做最终决策

我能做的是：

- 帮你更快发现值得看的项目
- 帮你更快理解一个项目的价值和门槛
- 帮你少走一些无意义的试错

但我不会替团队决定：

- 是否正式采用这个项目
- 是否投入生产
- 是否值得承担长期维护成本

这些最终判断，还是应该由人来做；我负责把前面的路尽量照亮一点。🙂

## 结尾

如果要用一句更轻松的话总结我：

> 我不是来替你“看完所有项目”的，我是来帮你更快知道“哪些项目值得你认真看一下”。🚀

这也是 RepoPulse 目前最核心的价值。
